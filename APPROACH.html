How AI Coding Agents Fix Bugs Over Time — Study Approach

What this study is. It is a follow-up to our MSR'26 paper, "Understanding the Rejection of Fixes Generated by Agentic Pull Requests." That paper took a single snapshot in time and asked why bug-fix pull requests (PRs) written by AI coding agents get rejected. This study keeps the same question but adds two things: a time axis — how does the picture change month by month? — and a wider scope, moving from one question to eight, all over a 15-month window.

The data. 422,618 bug-fix PRs opened between December 2024 and February 2026, drawn from the GitHub-Agentic-PR-Dataset. Of these, 121,832 were written by AI agents (Claude Code, Cursor, Copilot, Devin) and 300,786 were written by human developers.

The main idea: compare like with like

AI agents and human developers tend to work on different projects. So the obvious headline comparison — "agent fixes are rejected X% of the time, human fixes Y%" — is misleading, because it mixes up two separate things: who wrote the fix and which project the fix went into. Some projects reject far more PRs than others, no matter who wrote them.

We checked that this really matters. In projects where both agents and humans contribute, the rejection rates are close: 18.1% for agents and 15.1% for humans. In projects where only one of them contributes, both rates fall to 14.7% and 10.9%. The project clearly shapes the outcome.

So, to take the project out of the comparison, every agent-vs-human question is answered only inside the 1,218 projects where both agents and humans submitted bug-fix PRs (47,925 agent fixes and 287,358 human fixes). Inside those shared projects, any remaining difference is far more likely to reflect the author than the project. Questions that are only about agents (for example, "do people switch agents?") use the full agent dataset.

When does a fix count as "rejected"?

We use deliberately strict, simple rules so an outcome is never ambiguous:

  • A PR counts as rejected only if it was closed and never merged.
  • The rejection rate is measured only over PRs that have a final decision (merged or closed). PRs that are still open are left out, so undecided cases cannot distort the rate.

The eight research questions

"Matched" means the question is answered inside the 1,218 shared projects; "all agents" means it uses the full agent dataset.

How we prepare the data

From the raw PRs we derive a small number of clean signals:

  • Month. Each PR is placed in a month using its creation date; PRs with no date are dropped.
  • Outcome. Each PR is marked merged or rejected; blank or placeholder timestamps ("NaT", "None") are treated as not merged.
  • Fix size. Measured as lines added + lines removed ("code churn"). PRs with no recorded file changes are left out of the size results rather than counted as zero.
  • Tests and instruction files. Text patterns flag which PRs touch test files and which add agent instruction files (files that tell an agent how to work in a repo).
  • Linked issues. Each fix is connected to the issue it closes (phrases like fix/close/resolve #123) so we can follow what happens after a rejection (RQ2b).
  • Reverts. Detected by looking for reverts commit <id> in commit messages.

Full step-by-step details are in METHODOLOGY.md.

How we check the results are real

  • Trends over time are tested with the Spearman rank correlation — does the line move consistently up or down, rather than wandering by chance?
  • The effect of adding a test (RQ6) is tested with a χ² test, and then re-checked separately within groups of similar fix size. This confirms the effect is real and not just a side-effect of fixes being larger or smaller.

What this study cannot tell us

  • Agents and humans are compared only inside the 1,218 shared projects — this is by design, but it means the comparison does not cover projects where only one of them works.
  • RQ2b can only see the roughly 18% of fixes that explicitly reference an issue.
  • Revert counts are a lower bound: only reverts that go through a PR are visible.
  • RQ5 bug categories come from keywords and should be spot-checked by hand before publishing.
  • The overall ~16% rejection rate is not directly comparable to the earlier paper's snapshot — it comes from a much larger, more recent re-collection of data.
