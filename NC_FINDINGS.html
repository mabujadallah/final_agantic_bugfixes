How AI Coding Agents Fix Bugs Over Time — Nine New Findings

What this page is. Beyond the eight research questions described in the study approach, we ran nine further analyses (NC1–NC9, "new contributions") on the same dataset: 422,618 bug-fix pull requests (PRs) from December 2024 to February 2026 — 121,832 written by AI agents (Claude Code, Cursor, Copilot, Devin) and 300,786 by human developers. As before, every agent-vs-human comparison is made only inside projects where both contribute, so we compare like with like.

NC1 — Do instruction files help agents?

Developers can add an instruction file (such as CLAUDE.md or copilot-instructions.md) that tells an agent how to work in their project. We compared 351 projects before and after they adopted one, against 500 similar projects that never did — the standard "difference-in-differences" design, which removes the effect of time itself and of each project's own habits.

The result is a null: after adoption, rejection moves by +1.4 points, with an uncertainty range (−1.2 to +4.0) that comfortably includes zero. What does change is the work itself: after adoption, agent fixes become significantly larger and include tests more often. Instruction files appear to change how agents work, not how often their work is accepted.

NC2 — Can a wasted fix be predicted in advance?

Every rejected PR is wasted effort for the maintainer who reviews it. We trained a model on fixes from the past and asked it to predict rejections in a future it had never seen (training on data up to October 2025, testing on October–December 2025). It scores 0.76 on a scale where 0.5 is a coin toss and 1.0 is perfect — good enough to be useful for triage, e.g. flagging risky PRs for closer review.

The strongest signal is not the fix itself but the project's own history: how often that project rejected agent fixes before. Next come which agent wrote the fix and how long its description is. Whether the fix includes a test barely matters.

NC3 — How autonomous are agents really?

A merged agent PR does not necessarily mean the agent did all the work: a human may have pushed follow-up commits to finish the job. We counted, for every merged agent fix, whether any commit in it was authored by a human. This is only measurable for the cloud agents (Copilot, Devin), which commit under their own bot identity; Cursor and Claude Code run on the developer's machine and commit under the developer's name, so their fixes cannot be separated this way.

81% of merged cloud-agent fixes contain zero human commits (Copilot 78%, Devin 90%). Put the other way: about 1 in 5 "agent" fixes needed a human to step in before it could be merged. Autonomy is lowest on network and concurrency bugs (~78%) and highest on UI and authentication fixes (~86%).

NC4 — Did agents change, or did their tasks change?

Earlier we saw that Claude Code's rejection rate fell during 2025 while Copilot's rose. But that could be an illusion: maybe Claude Code was simply handed easier bugs later on. We split each agent's change into two parts: the part explained by a shift in task mix (bug type and fix size) and the part that remains at a fixed task mix — the real change.

The changes are mostly real. Claude Code improved by 5.8 points, of which 4.4 is genuine; Copilot worsened by 17.4 points, of which 18.3 is genuine (its task mix actually got slightly easier); Devin's 12-point improvement is also mostly genuine.

NC5 — Are agent fixes rubber-stamped?

Inside the same projects, agent fixes are decided much faster than human ones: the median agent fix is merged in 1.5 hours, against 6.3 hours for a human fix. One in five agent merges (21%) happens within 10 minutes of the PR being opened — too fast for any real review — versus 9% for human fixes.

The natural worry is that this is getting worse: that maintainers trust agents more and check them less. The data says the opposite. The share of instant merges is falling (21% → 16% over the window) and the median decision time is slowly rising. The sharpest correction is Copilot's: its instant-merge share collapsed from 22% to 8%. Scrutiny of agent code is increasing, not eroding.

NC6 — First impressions decide adoption

For each of 8,987 projects we found its first ever agent bug-fix PR and asked: did the project use an agent again within 90 days? If the first fix was accepted, 60% of projects came back. If it was rejected, only 40% did.

The effect differs sharply by agent. Devin suffers most: after a rejected first fix, only 26% of projects try again (versus 59% after an accepted one). Claude Code is nearly immune — 75% return even after a rejection, barely below the 79% after a success — suggesting its users are committed to the tool rather than evaluating it on a single trial.

NC7 — Do agent fixes hold up?

A fix that gets merged but doesn't actually solve the problem will soon be followed by another fix to the same code. So we measured: after a merged fix, how often does a later bug-fix PR touch the same source file within 90 days?

Measured naively over all files, both agents and humans land at 77% — but that number mostly reflects busy files that change all the time anyway. The honest comparison uses quiet files (files rarely touched by fixes), where a re-touch genuinely suggests the bug came back. There, agent fixes are re-fixed 29.5% of the time and human fixes 29.9% — statistically identical. Agent fixes last just as long as human fixes. This replaces our weaker revert-based measure (RQ4), where events were too rare (~0.1%) to support a confident comparison.

NC8 — The fixes that get no answer

Our rejection rate only counts PRs that received a decision. But some PRs are simply ignored — never merged, never closed. Giving every PR exactly the same 60-day observation window, we find 4.4% of agent fixes and 5.0% of human fixes still undecided after 60 days. Modest — but the agent share quadrupled over the study window, from about 2% to about 8%, with Copilot the most-ignored agent (11% overall).

This matters for reading the headline numbers: if ignored PRs were counted as rejections, agent rejection would rise from 16.9% to 20.6%. The seemingly flat rejection trend hides a growing pile of unanswered agent work.

NC9 — When two agents share a project, who wins?

In 526 projects, at least two different agents submitted bug fixes. Do such projects keep experimenting, or settle on a favourite? They settle: among projects where two agents genuinely competed early on, the agent mix becomes markedly more concentrated over time, and 36% end up using a single agent exclusively.

Head-to-head, Claude Code is the most common keeper: it wins 65% of projects it shared with Copilot and 81% of those shared with Devin. Devin loses every matchup (12–21%). And the choice is partly earned: in projects where two agents both had enough early fixes to compare, the agent with the lower early rejection rate ended up as the keeper 59% of the time — more than chance.

Read with care

  • NC1 finds no effect on acceptance; instruction files may still help in ways we cannot see here (fewer retries, better code style, happier developers).
  • NC3 covers only the two cloud agents; local agents (Cursor, Claude Code) blend into the developer's own commits by design.
  • NC5 cannot distinguish a self-merge from a reviewer merging instantly.
  • NC6 is observational, not a controlled experiment.
  • NC7's all-files number is dominated by file churn; rely on the quiet-files comparison.
  • NC9's smallest match-ups rest on few projects.
